# Snap Grid Recovery and Autoresearch Report

## Goal

Improve `snap()` so that:

- already-snapped images remain stable on repeated runs
- square pixel art generated by LLMs does not collapse into mismatched X/Y grid counts
- `fast-pixelizer` can ship a `1.2.0` release with stronger snap-mode behavior

## Problem Summary

The current `1.1.0` implementation reproduces two concrete failures on the sample images in `examples/`:

| File                   | Input size  | Current result                           |
| ---------------------- | ----------- | ---------------------------------------- |
| `1.gemini.png`         | `2048x2048` | `detectedResolution=201`, `grid=201x201` |
| `2.well-converted.png` | `2010x2010` | `detectedResolution=67`, `grid=65x69`    |
| `3.gpt.png`            | `1024x1024` | `detectedResolution=149`, `grid=148x150` |
| `4.gpt.png`            | `1024x1024` | `detectedResolution=139`, `grid=98x180`  |

The important regressions are:

1. `2.well-converted.png` is effectively the already-clean result of the first case, but a second `snap()` run collapses the detected grid from about `201x201` to about `67x67`.
2. `4.gpt.png` is a square image that should yield close X/Y grid counts, but the current detector diverges into `98x180`.

The GPT samples also reveal a second limitation of the input itself: these images are often "pixel-art-like" rather than true scaled pixel art, so the source already contains uneven cell widths and heights before any repair pass starts.

## Root Cause

Current `snap()` has two structural weaknesses:

1. **Step estimation depends on thresholded local peaks.**
   - When a clean image has many same-color adjacent cells, boundary peaks become sparse.
   - `estimateStep()` can return `null`, which sends the algorithm to the coarse `FALLBACK_SEGMENTS=64` path.
   - That is why `2.well-converted.png` falls back to a step of about `31px` and lands near `67x67`.

2. **Boundary walking is greedy and accumulates drift.**
   - `walk()` predicts the next cut from the previous snapped cut.
   - On noisy LLM-generated pixel art, local peak choices drift over time.
   - That drift is how a square image can end up with heavily mismatched axis counts such as `98x180`.

3. **Some ChatGPT-generated inputs do not contain a clean recoverable lattice.**
   - The image may look pixelated to a human, but the source can already contain inconsistent cell aspect ratios.
   - `snap()` can impose a square output lattice, but it cannot reconstruct a perfect original grid if the source never had one.
   - For generation workflows, Nano Banana is a better recommended source because it tends to preserve more stable square-grid structure in practice.

## Autoresearch Adaptation

I downloaded `karpathy/autoresearch` to `.tmp/autoresearch` and used its core workflow, not its GPU training loop:

- single in-scope implementation file: `src/snap.ts`
- fixed evaluation harness: `scripts/evaluate-snap-regressions.mjs`
- keep/discard loop driven by a stable metric instead of manual eyeballing

This mirrors the repo's design:

- one file is edited repeatedly
- one evaluation command is treated as the source of truth
- each experiment is accepted only if the score improves without adding unnecessary complexity

For this repo the evaluation metric is not `val_bpb`; it is a regression score built from:

- Gemini stability: first snap vs already-snapped re-run should stay aligned
- GPT square-grid consistency: square inputs should keep X/Y grid counts close

## Implemented Fix

1. Replaced fragile peak-only step estimation with an autocorrelation-based periodicity detector.
2. Kept rough cut discovery, but only to infer grid counts; final sampling now uses a regularized uniform lattice so local drift cannot compound across the image.
3. Added a fixed evaluation harness in `scripts/evaluate-snap-regressions.mjs`.
4. Added regression tests covering Gemini repeat-snap stability and GPT square-grid consistency.
